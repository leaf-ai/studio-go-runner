Platform Services - Overview

Objective

  • decide a useful path forward to create, deploy, and operate platform services


Lessons learned

  • service communications require a solid ecosystem
  • lack of infrastructure transparency has created widespread and large risks
  • devops blocker for stats, metrics, logging ... creating technology gaps


Requirements

  • flexibility to employ multiple styles, languages, and classes of services
  • address needs of multiple Sentient products
  • 1 person can deal with it

others too have experienced this. See more at
Monolith to ServiceMesh - Matt Klien, Lyft

What we did

stand up multiple skeleton services using, github.com/SentientTechnologies/platform-services and evaluate

  • HTTP/1.x & json   vs   HTTP/2 & proto
  • lights-out ops of kubernetes, using kops
  • reducing devops overhead, increase transparency using Istio
  • multi-language use of gRPC
  • AWS Aurora Postgres egress
  • roadmap to AWS EKS

How does it work

  • kubernetes is a deployment tool, data plane
  • istio is a service orchestration tool, currently people, control plane
  • Kubernetes is the mechanisim, Istio brings the policy
  • istio encapsulates entire suites of tools Netflix OSS Hystrix, Zuul, Ribbon, Eureka Zipkin, Atlas
  • istio stops unruly mashups of applications and network functions
